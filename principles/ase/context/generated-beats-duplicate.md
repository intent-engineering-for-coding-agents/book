# ASE-CONTEXT-GENERATED-BEATS-DUPLICATE: Generated Vendor Files Beat Authored Duplicates

**Layer**: 1
**Categories**: context, vendor-files, maintenance
**Applies-to**: all
**Summary**: A generated vendor file is a known point-in-time output; an authored duplicate is already behind.

## Principle

A generated vendor file is a known point-in-time output from a known process. An authored vendor file is a copy that is already behind by the time it is committed. Prefer generation over duplication for all vendor-specific instruction files.

## Why it matters

When the canonical instruction changes, every authored duplicate must be manually updated. One will be missed. The agent that loads the missed file gets stale instructions. Generated files eliminate the manual update step — regenerate all vendor files from the canonical source.

## Violations to detect

- Vendor-specific files with content matching `.agents/` files but different wording
- Vendor files that exist without a corresponding generation script or CI step
- Manually authored vendor files when a generation mechanism exists

## Good practice

The canonical instructions live in `.agents/`. A CI script or `dot-scout` generates `.github/instructions/`, `.claude/rules/`, and other vendor files from the canonical source. Committed generated files have a watermark: `<!-- generated by ... — do not edit manually -->`.
